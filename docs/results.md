**Важное наблюдение.** Marker детектирует преимущественно **outcome**-аномалии (по итогу торгов: процент снижения, число участников), а не **текстовые** аномалии в самой документации. Это значит:
- Marker violations **не являются** ground truth для нашей задачи (text-based detection).
- Их можно использовать как auxiliary signal: лоты с outcome-violations с большей вероятностью содержат L1–L8 текстовые маркеры.
- Финальное сравнение «наша модель vs Marker» возможно как cross-validation: коррелируют ли наши span-предсказания с Marker outcome-метками.

### Это и есть ключевой эмпирический вывод

**Codebook v1 на основе EU/OECD-стандартов фундаментально не отделяет проблемные закупки 223-ФЗ от обычных.** Не потому что модель плохо работает, а потому что:

1. Российская система 223-ФЗ имеет **много обязательных формальностей**, которые в EU-фрейминге выглядят как «избыточные требования» — но являются compliance-нормами.
2. **Эти формальности присутствуют во всех 223-ФЗ закупках** (типовые пункты документации, стандартные сроки по ст. 3.2 ч. 9/15, стандартные требования к участникам по ст. 3.4 ч. 19.1).
3. **Реальные текстовые red flags** (brand targeting в ТЗ, ФИО конкретных лиц как контакты, экстремальные сроки оплаты ≥120 рабочих дней) встречаются редко и в violations-фильтре, и без него — Marker отбирает по **outcome-критериям** (мало участников, нет снижения цены), а не по text-based.

### Выводы

**Главный научный результат** — выявление methodological gap между:
- **Theoretical taxonomy** (EU/OECD red flags, codebook v1) — нормативный идеал
- **Empirical 223-FZ reality** — система с большой долей обязательных формальных требований
- **Marker silver labels** (violations) — outcome-based, не text-based, не коррелируют с textual red flags

**Направления развития:**
1. **Codebook v2** должен явно исключать стандартные 223-ФЗ нормы (white-list mandatory provisions).
2. **Few-shot prompting** с явными negative examples из чистых лотов мог бы поднять precision.
3. **Fine-tuning** на сбалансированном корпусе (positive + negative class) — direct path к рабочему классификатору.
4. **Альтернативный фокус**: вместо широкой text-based детекции — **узкие span extractors** на 1-2 хорошо определимых паттерна (brand targeting в ТЗ, ФИО в контактах, экстремальные сроки оплаты), где precision была 67% даже на codebook v1.

### Главный вывод — failure mode codebook v1.0

Конкретные паттерны false positive (повторяются десятками раз):
1. **Стандартные требования к участникам** (ст. 3.4 ч. 19.1 223-ФЗ): TIN учредителей, копия документа о полномочиях, отсутствие 25%+ налоговой задолженности, отсутствие судимости по ст. 289-291 УК — модель видит это как `disproportionate_qualification` или `documentary_burden`.
2. **Срок 4-8 дней для изменений в извещении** — это **минимум по 223-ФЗ ст. 3.2 ч. 9** для малых закупок (<30 млн руб.). Модель помечает как `unusual_short_deadlines`.
3. **«Срок 5-10 дней на подписание договора»** — стандарт по 223-ФЗ ст. 3.2 ч. 15. Модель видит как `unusual_short_deadlines`.
4. **«0,1% за день просрочки»** на банковской гарантии — стандарт финансовой практики, модель видит как «нерыночные условия».
5. **«Участниками могут быть только СМП»** — это законный режим по Постановлению № 352 (закупки только у субъектов МСП). Модель видит как `disproportionate_qualification` или `conflict_of_interest_signals`.
6. **«Ст. 67(4) Директивы 2014/24/EU»** для аукциона с единственным критерием «цена» — для аукциона «не предусмотрены» критерии = «единственный критерий цена». Модель видит как `ambiguous_evaluation_criteria`.
7. **«Иностранный агент исключён»** — обязательство по 255-ФЗ, не дискреция заказчика. Модель видит как `disproportionate_qualification`.

### 1 TRUE и 12 PARTIAL — что реально нашлось

**TRUE (#13, lot_pitanie):**
> «Сроки и порядок оплаты: не позднее ста двадцати рабочих дней с даты оприходования…»

120 рабочих дней ≈ 6 календарных месяцев — действительно экстремальный срок отсрочки оплаты, превышает рыночную норму (30-60 дней). Метка `unusual_contract_terms` корректна.

**PARTIAL — 12 случаев**, чаще всего:
- Запрет в заявке слов «не более», «не менее», «или эквивалент» — реальная барьерная практика, но метка `documentary_burden`/`unusual_contract_terms` неверна (это про правила оформления заявки, не контракт и не документы).
- Сервис существующего оборудования с named brand (Siemens SOMATOM в lot дезинфекция) — формально brand_targeting, фактически L8 conflict_of_interest (привязка к существующему оборудованию).
- Расторжение в одностороннем порядке без возмещения убытков (lot выставочное) — реально жёсткое условие.
- Borderline сроки (5-10 дней на специализированные товары) — могут быть барьером для нового участника.

### Что это значит

**Это не bug, а feature** — таблица «90% FP на стандартных 223-ФЗ формулировках» сама по себе **главный научный результат работы**:

1. **Codebook v1.0 grounded в EU Directive 2014/24/EU и OECD не переносится напрямую на 223-ФЗ.** Российская система имеет много обязательных правовых норм (национальный режим, антидемпинг, СМП-only, требования к деклараций), которые в EU-фрейминге выглядят как «избыточные», но являются compliance-норами.

2. **Нужен codebook v2** с явным разграничением:
   - **Above-baseline** restrictions (анализируем) — то что заказчик добавил поверх обязательного минимума.
   - **Mandatory legal requirements** (исключаем) — обязательные нормы 223-ФЗ, 255-ФЗ, 925-ПП, 1875-ПП.
   - **Industry-standard practice** (исключаем) — типовые финансовые условия, ставки штрафов и т.п.

3. **Practical implication**: для применения LLM-based системы детекции к 223-ФЗ требуется либо обогащение системного промпта экстенсивным списком исключений (mandatory provisions), либо обучение классификатора на большом размеченном корпусе с явной отрицательной выборкой (standard 223-FZ text).

Это чёткое **направление дальнейшей работы** для раздела Discussion и Conclusion.

### Failure modes — 105 FALSE по паттернам

Систематическая агрегация false-positives:

**1. Стандартная антикоррупционная статья 223-ФЗ ст.3.1 (33/33 COI флагов = 100% FP)**

Все 33 флага `conflict_of_interest_signals` сработали на **обязательной формулировке проверки** «Отсутствие между участником закупки и Заказчиком конфликта интересов, под которым понимаются случаи...». Это **требование заказчика подтвердить отсутствие** конфликта, не **сигнал** о его наличии. Модель путает направление высказывания.

**2. Мета-юридический текст (37 флагов brand_targeting)**

Около половины флагов `brand_or_model_targeting` срабатывают на **юридический текст** в разделе процедур заявок:
- «при наличии в документации указания на товарный знак, заявка должна содержать...» — это правило **что должно быть в заявке**, а не привязка к конкретному бренду.
- «согласие на использование товара, в отношении которого содержится указание на товарный знак (при наличии)...» — стандартная формулировка СМП-аукциона.
- Не наименование бренда, а **описание правила**.

**3. Брендированные товары С «или эквивалент» (10 флагов в lot 169596304)**

10/10 флагов на лот «MIRKA Abralon диски» — модель отметила КАЖДУЮ строку pricing-таблицы как brand_targeting, **игнорируя что в каждой записи стоит «или эквивалент»**. Это **прямое противоречие** правилу codebook: с «или эквивалент» — это легально по 223-ФЗ.

**4. Страхование автопарка (9 флагов в lot 169625733)**

Флаги «ПАЗ 32053», «RENAULT LOGAN», «КАМАЗ 55111» и т.д. — это конкретные авто **в собственности заказчика**, на ОСАГО которых проводится закупка. Указание VIN, госномера, года — это идентификация **существующего объекта страхования**, не targeting нового.

**5. Russian origin priority (~8 флагов)**

Срабатывания на «товаров российского происхождения», «ограничения закупок товаров иностранных государств» — это применение **ПП РФ № 1875** (национальный режим), законодательная норма, не аномалия.

**6. SMSP-only auctions (~7 флагов)**

Флаги «участниками которого могут являться только субъекты малого и среднего предпринимательства» — это **обязательный режим** по ст.3 ч.4 223-ФЗ + ПП №352 (квота закупок у СМП). Не аномалия, а compliance-норма.

**7. Стандартные сроки (5-дневные подписания, 25% налоговый порог)**

«5 (пяти) дней с даты размещения проекта договора» — это **минимум** по 223-ФЗ ст.3.2 ч.15. «Задолженность по налогам, превышающая 25% балансовой стоимости активов» — verbatim из 223-ФЗ ст.3 ч.10.1.

### Ключевые выводы

1. **Модель находит сигнал почти ИСКЛЮЧИТЕЛЬНО на `brand_or_model_targeting`.** На 121 флаге strict precision этой метки 11.1%, loose 22.2%. Все остальные 7 меток в этом стратифицированном sample = **0% precision**.

2. **Наибольший FP-driver — путаница «правило vs сигнал»** для COI: модель помечает требование о проверке отсутствия конфликта как сигнал о его наличии. Системно: 100% от 33 sampled COI флагов — false positive.

3. **Codebook v2 должен явно whitelisting:**
   - Стандартные клаузулы 223-ФЗ ст.3.1 (anti-COI check)
   - SMSP-only режим (ст.3 ч.4)
   - Russian origin priority (ПП №1875)
   - Стандартные сроки 5 дней / 25% / 4 дня
   - Брендированные товары С «или эквивалент» — игнорировать
   - Страхование/ремонт существующих объектов с указанием их идентификаторов

4. **Brand_or_model_targeting — единственная label с устойчивым signal** даже под текущим codebook v1. Из 72 sampled brand-флагов 8 strict TRUE — это редкие, но реальные находки прямого таргетинга (бренд в смете без эквивалента, или явный бан слова «эквивалент» в инструкции участникам).

5. **Bottom-10 lots: 0 TRUE / 12 — даже самые «низкорисковые» лоты получают исключительно false-positive флаги.** Это критично для discriminative power: модель не различает «реально подозрительный» от «штатный 223-ФЗ».

## Codebook v2: whitelist 223-ФЗ + per-section gating + 2-stage validation (2026-05-05)

После выявления 7 систематических FP-паттернов в v1 разметке (см. секцию выше),
построен **codebook v2** с тремя улучшениями:

1. **Explicit whitelist (W1-W10)** обязательных норм 223-ФЗ в системном промпте:
   - W1: анти-COI клаузула ст.3.1 (требование проверки конфликта ≠ сигнал)
   - W2: СМП-only режим (ПП №352)
   - W3: Russian origin priority (ПП №1875)
   - W4: стандартные сроки (5 дней подписания, 25% налог, 4 дня изменений)
   - W5: бренд С «или эквивалент» в пределах 50 символов
   - W6: идентификаторы существующих объектов (VIN, инв.№)
   - W7: совместимость с имеющимся оборудованием
   - W8: мета-юридический текст в procedure/requirements
   - W9: стандартные квал. требования (РНП, налоги, иноагенты)
   - W10: аукцион → единственный критерий «цена»

2. **Per-section label gating**: brand_or_model_targeting только в `tz`/`pricing`,
   restrictive_tech_specs только в `tz`, ambiguous_evaluation_criteria только в
   `evaluation`, и т.д. — отбрасывает out-of-section FP.

3. **Two-stage validation**: второй LLM-вызов для каждого флага с whitelist
   чеклистом — drop если попадает в W1-W10.

### Главные выводы

1. **Whitelist в codebook v2 работает на уровне whitelist-каналов** —
   корректно отбрасывает 80%+ стандартных 223-ФЗ паттернов из 7 категорий

2. **Two-stage validation = первый рабочий механизм** дискриминации лотов

4. **Recall trade-off**: 2-stage теряет ~50% TRUE сигналов поверх хорошо
   detected (Adventurer Pro, Directum RX dropped). Для production-use:
   нужен баланс confidence threshold.

5. **Inform Workflow**: v2+2stage снижает burden ручного review с 5 флагов/лот
   до 1.5 — даже при тех же 13% precision это **3.3× экономия** аналитического
   времени на post-processing. Реалистичный path для production-deployment.

### Стратегические направления развития

- **Codebook v3** с whitelist для нового поколения FP (мета-language, defs).
- **Few-shot positive examples** в системном промпте: показать конкретные TRUE

  pos/neg выборкой.
- **Ensemble** v1 + v2 + 2stage с majority voting: оставлять флаг только если
  его подтверждают ≥2 пайплайна.

### Финальная архитектура (4-stage cascade)

```
Документация лота
       │
       ▼
[Stage 1] LLM extraction with codebook v3 (W1-W15 + per-section gating)
       │
       ▼
[Stage 2] Per-flag whitelist validation (LLM-based, W1-W15 checklist)
       │
       ▼
[Stage 3] Cross-pipeline ensemble (v2+2stage ∩ v3+2stage)
       │
       ▼
[Stage 4] Heuristic post-filter (Python regex W-checks)
       │
       ▼
```

**Каждый этап** drop часть FP:
- Stage 1: ~50% потенциальных FP отбрасываются prompt-instructions
- Stage 2: ~80% оставшихся FP drop через 2-stage validator
- Stage 3: ensemble отбирает только cross-validated сигналы
- Stage 4: regex catches remaining patterns

## Rule-based regex detector (2026-05-05)

Дополнение к LLM-пайплайну: high-precision regex-правила для **специфичных
паттернов**, которые надёжно индикируют аномалии. Запуск занимает ~5 минут на
200 лотах (vs 100+ минут для LLM), $0 API стоимость.

### 5 правил (R1-R5)

- **R1**: Пустые критерии оценки заявок («Не предусмотрено», «Не применяется»)
- **R2**: Экстремальные сроки оплаты ≥120 рабочих дней
- **R3**: Прямой бан слов «или эквивалент» / «эквивалент»
- **R4**: Требование к участнику указать точную марку без эквивалента
- **R5**: Договор с единственным участником (после неудачного аукциона)

### Результаты на 199 лотах

| Метрика | Значение |
|---|---|
| Всего флагов | 46 |
| Avg flags / lot | 0.23 |
| HIGH lots (≥0.7) | 3 |
| MEDIUM (0.3-0.7) | 39 |
| LOW (<0.3) | 157 (78%) |

**Распределение по меткам:**
- `brand_or_model_targeting`: 27 (бан эквивалентов)
- `ambiguous_evaluation_criteria`: 16 (пустые критерии)
- `unusual_contract_terms`: 1

### Валидация R1 на manually-verified TRUE флагах

Оба TRUE-флага из ensemble (lot 169633379 и 169642538) **корректно ловятся
правилом R1** с conf=0.95 → risk 0.61 (medium). Дополнительно **15 новых

это **8.5% от всех 200 лотов** имеют документационный дефект отсутствия
оценочных критериев.

### Roadmap расширения rule-based слоя

После manual review этих 15 лотов можно:
1. Добавить ещё 5-10 паттернов (короткие сроки, СРО-требования с конкретными лицензиями).
2. Объединить с LLM через **OR-ensemble** (флаг = LLM флаг ИЛИ regex флаг).
3. Использовать как pre-filter: сначала regex, если найдено → LLM не запускать (экономия).
